---
title: Schema discovery
---

`GET lead_schema` returns, for your key: the exact fields you may send and read, which are required, your routing mode, and how each child table works.

:::note{title="Discover, don't hardcode"}

Two keys get different schemas, and grants change over time. Read `lead_schema` at startup and build your payloads from its response — never copy a fixed field list from this page.

:::

```bash
curl -H 'Authorization: token <api_key>:<api_secret>' \
     https://one.tatvacare.in/api/method/tatva_connect.api.partner.lead_schema
```

## What comes back

The response is top-level `status` + `data`. Here's a trimmed example for a key granted a few lab and plan fields:

```json
{
  "status": "success",
  "data": {
    "lead": [
      { "fieldname": "mobile_no",  "label": "Mobile No",  "type": "Data", "required": false, "options": null },
      { "fieldname": "first_name", "label": "First Name", "type": "Data", "required": false, "options": null },
      { "fieldname": "status",     "label": "Status",     "type": "Link", "required": true,  "options": "CRM Lead Status" }
    ],
    "children": {
      "custom_lab_profile": {
        "multi_row": true,
        "key_field": "report_date",
        "fields": [
          { "fieldname": "hba1c",       "label": "HbA1c",       "type": "Float", "required": false, "options": null },
          { "fieldname": "report_date", "label": "Report Date", "type": "Date",  "required": true,  "options": null }
        ]
      },
      "custom_plan_profile": {
        "multi_row": false,
        "key_field": null,
        "fields": [
          { "fieldname": "policy_number", "label": "Policy Number", "type": "Data", "required": false, "options": null }
        ]
      }
    },
    "dedup": "A lead is unique per (mobile_no, product line, group). Re-sending the same patient updates that lead. Program is NOT part of identity: sending the same patient with a different program transitions the SAME lead, never a new one.",
    "bulk": { "max_per_call": 100, "list_page_max": 200, "list_filters": ["status","created_after","created_before","updated_after","updated_before","mobile_no"] },
    "routing": {
      "mode": "list",
      "source": "…", "vertical": "…", "group": "…",
      "program": null,
      "allowed_programs": ["Program A", "Program B", "Program C"],
      "program_required": true,
      "note": "Line and group are fixed. Send custom_current_program from allowed_programs on every lead. Program is a mutable attribute, not identity: the same patient on a new program is the same lead (a transition)."
    }
  }
}
```

## How to read each block

| Block | What it tells you |
|---|---|
| `data.lead[]` | Your top-level lead fields. Each carries `fieldname`, `label`, `type`, `required`, and (for Link/Select) the allowed `options`. |
| `data.children` | One entry per child table you can use — keyed by its array name (e.g. `custom_lab_profile`). |
| `data.children.<t>.multi_row` | `true` = the table holds many rows, addressed by a key. `false` = a single row that you merge into. |
| `data.children.<t>.key_field` | On a multi-row table, the field that addresses each row (the row's "primary key"). `null` on single-row tables. |
| `data.children.<t>.fields[]` | The fields you may send/read in that table — same shape as `lead[]`. The `key_field` is always present and `required`. |
| `data.routing.mode` | `forced` (one program stamped for you) · `list` (you must send `custom_current_program`, one of `allowed_programs`) · `none` (no program). Line and group are always fixed for you. |
| `data.routing.allowed_programs` | **List mode only** — the programs you may set per lead. `program_required: true` means a create must carry `custom_current_program`. |
| `data.dedup` | The rule that makes a lead unique: phone + product line + group. Program is an attribute, not identity. |
| `data.bulk` | Bulk caps and the filters `lead_list` accepts. |

### Each field's attributes

Every entry in `lead[]` (and in a child table's `fields[]`) describes one field:

| Attribute | Meaning |
|---|---|
| `fieldname` | the key you send/read in JSON — e.g. `mobile_no` |
| `label` | the human-readable name — e.g. "Mobile No" |
| `type` | the data type — `Data`, `Int`, `Float`, `Date`, `Link`, `Select`, … |
| `required` | `true` = must be present when you create the lead |
| `options` | for `Link`/`Select`, the allowed target or values; `null` otherwise |

## Required fields

A field with `"required": true` must be present on a create. Read this straight from the schema rather than assuming — your required set is whatever your key is granted, and it can differ from another partner's.

## Child tables: how rows are written

Child tables are sent as JSON arrays nested in the same lead body. There are two kinds, and the schema tells you which is which.

### Multi-row tables (`multi_row: true`) — upsert by key

Each row is addressed by its `key_field`. The write **merges by that key** — it never replaces the whole array:

- **New key** → a new row is appended.
- **Same key** → that row is updated, and **only the fields you sent change**. Omitted fields stay.
- **Rows you don't mention** → left untouched.
- **Delete one row** → send `{ "<key_field>": <value>, "_delete": true }`.

Every row you send **must** carry its `key_field` — it's the row's address. The schema marks it `required` even if it isn't in your field grant.

```json
// Update the 2026-05-18 lab row's hba1c, add a new row for 2026-06-20,
// and delete the 2026-01-10 row — all in one write. Other rows untouched.
{
  "mobile_no": "+919900000000",
  "custom_lab_profile": [
    { "report_date": "2026-05-18", "hba1c": 6.9 },
    { "report_date": "2026-06-20", "hba1c": 6.4, "fbs": 110 },
    { "report_date": "2026-01-10", "_delete": true }
  ]
}
```

### Single-row tables (`multi_row: false`) — merge the one row

Send a one-element array; the fields you send are merged onto the single row (created if it doesn't exist yet). Sending two distinct rows is ambiguous and returns a `validation_error`.

```json
{
  "mobile_no": "+919900000000",
  "custom_plan_profile": [
    { "policy_number": "NB-POL-0001" }
  ]
}
```

:::caution{title="Partial writes never clobber"}

Whether multi-row or single-row, sending only some fields updates only those fields. The rest of the row — and every row you didn't mention — is left exactly as it was. So you can safely send incremental updates as new data arrives.

:::

## Routing: line & group are fixed; program depends on your mode

Your `source`, product line (`vertical`), and `group` are fixed to your key and stamped on every lead automatically — you don't send them, and you can't move a lead to a different line or group. The **program** depends on your `mode`:

- **Forced** — stamped for you, like the rest of your routing.
- **List** — **you send** `custom_current_program` on every lead, validated to your `allowed_programs`. It is a mutable attribute: re-sending a patient with a different program transitions the **same lead** (a transition), it never makes a second one.
- **None** — no program.

## Reading rows back

When you read a lead (`lead_get` / `lead_get_bulk`), each child row carries its `name` — the row's unique ID, e.g. `a1b2c3` — and its `key_field`, so you always know how to address it on the next write.
