The small, cross-cutting rules that apply to every call.

## Auth

Every request carries your key in one header:

```
Authorization: token <api_key>:<api_secret>
```

Literal lowercase `token`, one space, then `api_key:api_secret` joined by a colon. A missing or malformed header returns `401 unauthorized`. Verify a key works with a single read:

```bash
curl -H 'Authorization: token <api_key>:<api_secret>' \
     https://one.tatvacare.in/api/method/tatva_connect.api.partner.lead_schema
```

A `200` with your schema means you're in. Contact **TatvaCare Ops** to request a key.

## Phone format

Phone numbers are E.164: `+<country code><digits>`, no spaces or hyphens — e.g. `+919999000123`. A bare 10-digit Indian number is auto-normalized to `+91…`, but sending the full E.164 form is the safe default.

## Idempotency & retry safety

| Call | Safe to retry? |
|---|---|
| `lead_create` | **Yes** — idempotent on phone + product line + group. The same input converges to the same lead; it never duplicates. |
| `lead_update` | **Yes** — same payload, same final state. Children merge by key; nothing is clobbered. |
| `lead_delete` | **Yes** — a second delete of a gone lead just returns `not_found`. |
| bulk variants | **Yes** — each record is by-key, so a replay converges. Re-send only the failed rows from `results[]`. |
| reads (`lead_get`, `lead_list`, `lead_get_bulk`) | **Yes**, always. |

Because writes are idempotent, the right response to a network hiccup or a `5xx` is simply to retry with backoff.

## Partial updates never clobber

A write touches **only the fields you send**. Omitted top-level fields keep their values; in child tables, omitted fields and unmentioned rows are left alone. Send incremental updates freely as new data arrives — see [Schema discovery](/discover-schema#child-tables-how-rows-are-written).

## Versioning

This API is `v1`. Backward-compatible changes — new fields, new endpoints, new option values, or a new field granted to your key — ship without a version bump and won't break your integration. Anything **breaking** (a field rename, an endpoint removal, tighter validation) gets advance notice by email to integration owners, with a migration window of at least four weeks.

Because your contract is whatever [`lead_schema`](/discover-schema) returns, code that reads the schema at runtime adapts to additive changes automatically.

## Environment

Production is `https://one.tatvacare.in`. There's no separate staging endpoint today. To test safely, use clearly-test data — e.g. a `mobile_no` like `+91999900xxxx` — which Ops periodically cleans up. Need a sandbox? Email Ops.

## Getting help

All key requests, rate-limit raises, new-field grants, and integration issues go to **TatvaCare Ops**.

When something's failing in an active integration, include:

1. The exact request — URL, method, body.
2. The full response — HTTP status and body.
3. The CRM `name` of any lead affected.
4. Timestamps (with timezone).
