---
title: Response shapes
---

Every call returns one of three top-level shapes. Knowing them up front means your client never has to guess.

## Success — a single record

`lead_create`, `lead_update`, `lead_delete`, `lead_get`:

```json
{
  "status": "success",
  "action": "created",
  "data": { "name": "CRM-LEAD-2026-00123", "source": "…", "vertical": "…", "group": "…", "program": "…" }
}
```

- `status` is `"success"`.
- `action` is `created` / `updated` / `deleted` (writes) or `fetched` (reads). It's absent on `lead_schema`.
- `data` is the result: the lead's unique ID (`name`) and your routing on a write, the curated lead on a `lead_get`, or your schema on `lead_schema`.

On a delete, `data` carries just `{ "name": "…" }`.

## Success — a list

`lead_list` returns a page object under `data`:

```json
{
  "status": "success",
  "action": "fetched",
  "data": { "total": 240, "count": 20, "offset": 0, "limit": 20, "leads": [ /* curated leads, no children */ ] }
}
```

`total` is the full count for your filters; `leads[]` is this page. Page with `offset` (max 200 per page). Lists omit child tables — use `lead_get` for a full record.

## Success — a bulk batch

Every bulk method returns **HTTP 200 even when some rows failed**. Read `summary`, then walk `results[]`:

```json
{
  "status": "success",
  "summary": { "total": 2, "succeeded": 1, "failed": 1 },
  "results": [
    { "index": 0, "status": "success", "action": "created", "name": "CRM-LEAD-2026-00125" },
    { "index": 1, "status": "error", "error": { "code": "validation_error", "message": "mobile_no is required" } }
  ]
}
```

- `summary` is `{total, succeeded, failed}` for writes, or `{requested, found, not_found}` for `lead_get_bulk`.
- `results[]` is **input-ordered** — `results[i]` is the outcome of the i-th item you sent.
- Each row carries its own `status`; a failed row carries the same `error` block as a single-call failure.

:::caution{title="A 200 doesn't mean every row landed"}

On bulk, always check each `results[].status`, not just the top-level one. One bad row reports its error while the rest still commit.

:::

## Failure — the error envelope

Any failure returns this exact shape, with a matching HTTP status:

```json
{ "status": "error", "error": { "code": "validation_error", "message": "…" } }
```

Branch on `error.code` (stable), show `error.message` to a human. Full details and the per-code playbook are on **[Errors](/partner-errors)**.

Successful calls are `200`; every failure carries one of a small set of stable `error.code` values. The full per-code table — HTTP status, meaning, and what to do — lives on **[Errors](/partner-errors)**. See [Reading responses](/reading-values) for pulling any value out by path.
