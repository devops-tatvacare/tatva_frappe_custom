This is the **integration-facing** API for TatvaCare's CRM. It exists so external systems — insurer feeds, the Product Line C mobile app, lab partners, and more to come — can push leads in and read state back. The CRM web app is where internal teams work those leads day to day; this API is the door for systems.

### Who reads this

- Engineers wiring an **inbound lead feed** into the CRM (e.g. an insurer sending lab-screened leads).
- Engineers wiring a **parallel-write** from a system that already creates leads elsewhere.
- Anyone who needs to **read** leads, their profiles, or their substages from outside the CRM.

### How these docs are organised

- **Documentation** (this tab) — getting started, Frappe concepts in plain English, the data model, the WhatsApp and telephony channels, and reference material (error codes, roles & permissions, tags).
- **API Reference** — every endpoint, grouped by resource (Leads, Files, Reference Data, Schema, Authentication), with a live try-it console.

### Scope today — and where it's growing

This spec focuses on **Lead**, the record every integrator touches. The CRM has other doctypes too — Tasks, Notes, Call Logs, Comments, and more — all reachable through the same Frappe REST pattern. We're building the documented surface out as integrations need it: if you need one that isn't covered yet, introspect it live (see [Schema discovery](/frappe-discovery)) or ask CRM Ops and we'll add it.

Two channel surfaces are already being built out — [WhatsApp](/whatsapp) and [Telephony](/telephony). Both follow the same "it's just a doctype" pattern: read a lead's messages or calls with a filtered list, exactly like everything else here.

### One reading rule

If a field, value, or behaviour isn't documented here, don't guess — use the [Schema](/api/schema) endpoint to introspect the live model, or email CRM Ops.
