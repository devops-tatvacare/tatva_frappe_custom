## Operational reference

### Rate limits

Frappe enforces a soft limit of ~25 requests per 5 seconds per API key by default. Bulk operations (backfill / migration scripts) should target **5 requests per second** to stay safe.

When you hit the limit, you get HTTP 429. **Always retry with exponential backoff** — 1s, then 2s, then 4s, etc., capped at ~30s. A simple retry loop:

```python
import time
for attempt in range(6):
    r = requests.post(...)
    if r.status_code != 429: break
    time.sleep(2 ** attempt)
```

If you need sustained throughput above this, email Ops — we can raise the limit per key.

### Retry safety per endpoint

| Endpoint | Safe to retry? |
|---|---|
| `POST .../partner.lead_create` | ✅ Yes — idempotent on the lead grain (phone + line + group). Same input = same final state. |
| `PUT .../partner.lead_update` | ✅ Yes — same payload = same final state. Profile children merge field-by-field (never replaced). |
| `POST/PUT/DELETE .../partner.lead_*_bulk` | ✅ Yes — each record is create-or-update / by-name, so a replay converges. Partial-success response reports any per-row errors. |
| `POST /api/method/upload_file` | ⚠️ Creates a new File row each time. Dedup client-side by checking `content_hash` if you re-upload. |
| `DELETE /api/resource/File/{name}` | ✅ Yes (404 on second call is fine). |
| All GETs | ✅ Yes, always. |

### Versioning policy

This spec is `v1.0.0`. Backwards-compatible changes (new fields on existing schemas, new endpoints, new lookup values) ship without version bumps. **Breaking** changes (field renames, endpoint removals, validation tightening) get advance notice to all integration owners via email, with a migration window of at least 4 weeks.

The `version` in the OpenAPI spec is the docs version. The CRM platform itself is Frappe v15 + Frappe CRM 1.71 today.
