This CRM has **two** API surfaces, and they speak **different** error envelopes. Know which one you're on before you write any error handling.

## Two contracts, by surface

| You're calling… | Envelope | Auth role |
|---|---|---|
| **Gated partner methods** — `tatva_connect.api.partner.lead_*` (`lead_create`, `lead_get`, `lead_list`, the bulk variants) | **The partner contract** — top-level `status` + `error.code`, no `message` wrapper, no `exc_type`. Full per-code table and playbook: **[Errors](/partner-errors)**. | Your partner key. |
| **Native Frappe REST** — `/api/resource/*`, `/api/method/<other>` | Frappe's native shape: success under `message`/`data`, failure as `{exc_type, exception, _server_messages}` (below). | Internal first-party keys (partner keys get 403 here). |

If you're integrating leads, you're on the **partner methods** — see [Errors](/partner-errors) for the envelope and the per-code playbook. The native shape below only matters for direct `/api/resource/*` calls (admin / back-office).

## Native Frappe errors (`/api/resource/*`)

Direct REST calls (internal first-party keys) return Frappe's native envelope:

```json
{
  "exc_type": "LinkValidationError",
  "exception": "Could not find CRM Program: Program BXXX",
  "_server_messages": "[\"Could not find CRM Program: Program BXXX\"]"
}
```

Check the HTTP status and read `exception`. This is **not** the partner contract — if you see `exc_type` while calling a `partner.*` method, file a bug.

### The ones internal keys hit most

- **`unauthorized` + an HTML login page** — Frappe falls back to the website login when auth fails on `/api/resource/*`. The header must be exactly `Authorization: token <api_key>:<api_secret>` — lowercase `token`, one space, colon-joined.
- **`LinkValidationError`** — a `Link` value doesn't exist in its master (e.g. an unknown program or source). Introspect the master with [Runtime discovery](/frappe-discovery) to enumerate valid values, then resend.
- **A value "didn't stick"** — a field above your key's permission level is silently dropped on write while the rest saves. It's a field-level lock, not a payload bug — see [Access control](/frappe-permissions).
- **`cannot_delete` / linked-document errors** — a lead with linked activity (calls, tasks, notes) can't be deleted.
