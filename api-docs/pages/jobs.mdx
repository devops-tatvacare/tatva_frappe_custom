---
title: Scheduled jobs
---

The CRM runs work on a clock (scheduled jobs, e.g. the 6-hourly WhatsApp template sync) and in the background (queued jobs, e.g. processing an inbound webhook). This page is the inventory — what exists, and how to list / inspect it over the API.

## The two doctypes you read

| Doctype | What it is |
|---|---|
| **`Scheduled Job Type`** | One row per scheduled job — its method, frequency/cron, whether it's stopped, and when it last ran. The **definitions**. |
| **`Scheduled Job Log`** | One row per execution — timestamp, status (`Scheduled` / `Complete` / `Failed`), and error details. The **history**. |

Both are normal doctypes, so you read them over the same REST API as everything else. **Auth:** `Authorization: token KEY:SECRET`, and you need an internal admin key (these aren't exposed to plain CRM users). There is **no REST endpoint to *trigger* a job** — running on demand is Desk- or bench-only (see below).

## List every scheduled job

```bash
curl -G -H 'Authorization: token KEY:SECRET' \
  --data-urlencode 'fields=["name","method","frequency","cron_format","stopped","last_execution"]' \
  --data-urlencode 'limit_page_length=0' \
  'https://one.tatvacare.in/api/resource/Scheduled Job Type'
```
- `stopped = 1` means the job is disabled.
- `last_execution` is null if it has never run (e.g. the scheduler is paused).

## Find one job (e.g. all the WATI ones)

```bash
curl -G -H 'Authorization: token KEY:SECRET' \
  --data-urlencode 'filters=[["method","like","%tatva_connect%"]]' \
  --data-urlencode 'fields=["name","method","cron_format","stopped","last_execution"]' \
  'https://one.tatvacare.in/api/resource/Scheduled Job Type'
```

Read one in full:
```bash
curl -H 'Authorization: token KEY:SECRET' \
  'https://one.tatvacare.in/api/resource/Scheduled Job Type/templates_sync.scheduled_sync_all'
```

## Did it actually run? — execution history

```bash
curl -G -H 'Authorization: token KEY:SECRET' \
  --data-urlencode 'filters=[["scheduled_job_type","=","templates_sync.scheduled_sync_all"]]' \
  --data-urlencode 'fields=["name","status","creation"]' \
  --data-urlencode 'order_by=creation desc' \
  --data-urlencode 'limit_page_length=20' \
  'https://one.tatvacare.in/api/resource/Scheduled Job Log'
```
`status`: `Complete` (ran fine), `Failed` (check the row's `details` for the traceback), `Scheduled` (queued).

## Is the scheduler even on?

A job only fires if the site's scheduler is enabled — and that's **not** a record you can read over REST. Check it from bench:
```bash
bench --site <site> scheduler status     # Enabled / Disabled
bench --site <site> scheduler resume     # turn it on  (alias: enable-scheduler)
bench --site <site> doctor               # health incl. scheduler heartbeat
```
Prod runs with the scheduler enabled. (Internally it's the `pause_scheduler` flag in the site config.)

## Run a job on demand (not via REST)

- **Desk UI:** `/app/scheduled-job-type` → open the job → run it; `/app/scheduled-job-log` shows the history.
- **bench:** runs the underlying method immediately, ignoring the schedule:
  ```bash
  bench --site <site> execute tatva_connect.whatsapp.templates_sync.scheduled_sync_all
  ```

## Current job inventory

| Method | Frequency | What it does |
|---|---|---|
| `tatva_connect.whatsapp.templates_sync.scheduled_sync_all` | every 6 h (`0 */6 * * *`) | Re-syncs every WATI account's approved templates into the local catalog, so the picker stays current. Respects the WATI kill-switch. |

Frappe also registers many **built-in** jobs (backups, email queue, search index, log cleanup, etc.) — they show up in the same `Scheduled Job Type` list. Filter by `method like "%tatva_connect%"` to see only ours.

## Background (queued) jobs — a quick note

Not everything is on a clock. Some work is pushed onto a **queue** with `frappe.enqueue` and run by a worker — for example, the **WATI inbound webhook** acks immediately and processes the message on the `short` queue. These are one-off jobs, not scheduled ones, and live in the **`RQ Job`** surface (Desk: `/app/rq-job`, or **Background Jobs** under Tools) rather than `Scheduled Job Type`.
