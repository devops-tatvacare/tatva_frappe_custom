One API to push leads into TatvaCare and read their state back. Your key decides which fields you may read and write, and which product line and group your leads belong to — there is no fixed payload.

:::tip{title="Your key is your contract"}

Your fields, which are required, and how the child tables work are all defined by your key. Discover them by calling **[`GET lead_schema`](/discover-schema)** — that response is your contract. Build from it; never hardcode a field list.

:::

## What you can do

- **Create or update leads** — by phone, idempotent, one at a time or up to 100 at once.
- **Read leads** — fetch one by id or phone, or list a page with simple filters.
- **Delete leads** — by id, single or bulk.
- **Discover your schema** — ask the API what your key may send and read, at runtime.

Every call is a single HTTPS request with a token header. Each endpoint has a live try-it console in the [Partner API Reference](/partner-api).

## The model in brief

- **One lead per patient** — identified by `phone + product line + group`. Re-sending updates that lead; it never duplicates, so every write is safe to retry.
- **Product line and group are fixed to your key** — you never send them.
- **Program is a mutable attribute, not identity** — your key sends it (List mode), has it stamped (Forced), or has none. The same patient on a new program stays the same lead.
- **Child tables** hold richer, repeating data — lab reports, plan details, the care team — sent as JSON arrays in the same body.

Full detail: [Lead identity](/concepts).
