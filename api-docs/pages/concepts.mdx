title: Lead identity
---

Four ideas cover the whole API.

## Identity tuple

A lead is identified by a **three-part key (a tuple)**:

| Part | Field | Meaning |
|---|---|---|
| Phone | `mobile_no` | the patient's number, **E.164** format (`+<country><digits>`, no spaces or hyphens) — e.g. `+919999000123` |
| Product line | `custom_vertical` | the line of business the lead belongs to |
| Group | `custom_group` | the unit within that line (e.g. a partner, or a program family) |

**One lead = one `phone + product line + group`.** Re-sending the same trio **updates** that lead — never a duplicate, so every write is safe to retry. The same phone under a **different** product line or group is a **separate** lead.

Product line and group are **fixed to your key** — you never send them; the API stamps them server-side.

## Program attribute

`custom_current_program` records the lead's current program. It is **not** part of identity: it can **change** on the same lead (a *transition*), and changing it never creates a second lead.

Whether you send it depends on your key's **program mode** — the only thing that differs between keys:

| Mode | Send `custom_current_program`? | Behavior |
|---|---|---|
| **Forced** | No | one program is stamped for you; anything you send is ignored |
| **List** | **Yes, required** | send one from your allowed set; it is validated. Re-sending with a different one transitions the same lead |
| **None** | No | no program |

Call [`lead_schema`](/discover-schema) to see your mode and, for **List**, your allowed programs.

## Child profiles

```
  CRM Lead  ─┬─→  Plan Profile               (subscription/PSP/drug/payment/acquisition)
             ├─→  Lab Profile                (lab values, vitals, conditions, scores)
             └─→  Care & Providers Profile   (coaches, RM, therapist, doctor, hospital)
```

Profiles are sliced by **concern**, not by org or program:
- A drug-program lead: Plan (drug-program details + Rx workflow) + Care & Providers (coach + doctor + hospital).
- An insurer lead: Plan (policy + payment) + Lab (lab values, vitals, conditions).
- A subscription lead: Plan (subscription) + Lab (metabolic score) + Care & Providers (coach + relationship manager + diet coach).

A lead on multiple programs simultaneously has **multiple child rows of the SAME type populated together** — multi-program is normal.

Profile children are **`istable=1`** in Frappe. That means:
- You CANNOT call `/api/resource/CRM Plan Profile` directly (returns 403).
- You read them embedded under the parent: `lead_get` returns `custom_plan_profile: [{...}]` inline.
- You write them embedded: send `custom_plan_profile: [{...}]` in your `lead_create` / `lead_update` body.
- `lead_create` / `lead_update` **merge** child rows field-by-field, so a partial write never wipes the row. (A raw REST `PUT` to `/api/resource/CRM Lead/<name>`, available to internal keys only, REPLACES the whole array — use [read-modify-write](/api/leads) there.)

## Substages

Each program has its own substage workflow — different stages, different naming. Today there are substage rows across all the active programs.

Instead of a separate substage column per program, we have ONE `CRM Plan Profile.substage` Link field pointing to a master `CRM Lead Substage`. Each row in the master is keyed `{program}-{substage_name}` — e.g. `Program A-Step One Done`, `Program B-Step Two Done`.

New programs add ROWS to that master, never new columns on Plan Profile. Enumerate the valid substages for a program via [Lookups → Lead Substage](/api/reference-data).

The **universal lead `status` field** (`New / Contacted / Nurture / Qualified / Converted / Junk / Unqualified`) is independent of substage. Substage is the program-specific operational layer; `status` is the coarse universal layer for cross-program reporting.
