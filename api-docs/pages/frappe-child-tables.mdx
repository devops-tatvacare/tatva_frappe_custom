---
title: "Child tables & Links"
---

Two field types connect DocTypes to each other — and they're the heart of how this CRM models a patient.

**`Link` = a foreign key.** A `Link` field stores the `name` (ID) of a document in *another* DocType. `CRM Lead.status` links to `CRM Lead Status`; `custom_current_program` links to `CRM Program`. The value must already exist in the target DocType, or the write is rejected.

**`Table` = a child table (rows inside a record).** A `Table` field holds a *list of sub-rows* that belong to the parent document. They have no independent life — they live and die with the parent.

In this CRM, the lead carries identity + contact + routing on the parent, and **everything program-specific lives in child "profile" tables**:

```json
{
  "first_name": "Asha Verma",
  "mobile_no": "+919800000099",
  "custom_lab_profile": [
    { "hba1c": 7.2, "fbs": 138, "report_date": "2026-05-18" }
  ]
}
```

`custom_lab_profile` is a child table — a list of rows. A lead can carry a Lab Profile, a Plan Profile, and a Care & Providers profile (each its own child table); a patient typically has 1–3 populated.

**Why it matters for you:**
- You **cannot** create a child row on its own — there's no `/api/resource/...` you POST a profile to (direct child-table access returns 403, by design). You send child rows *embedded in the parent* lead, exactly as above.
- A `Link` value must exist first. `custom_current_program: "Sigrima"` only works if `Sigrima` is a row in `CRM Program` — check via [Reference Data](/docs/api/reference-data).

→ **Try it:** [Upsert a lead by phone](/docs/api/leads#upsert-a-lead-by-phone-recommended) — its examples show profile child rows embedded in the body.
