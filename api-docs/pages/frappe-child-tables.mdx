title: Child tables
---

Two field types connect DocTypes to each other ‚Äî and they're the heart of how this CRM models a lead and its related records.

**`Link` = a foreign key.** A `Link` field stores the `name` (ID) of a document in *another* DocType. `CRM Lead.status` links to `CRM Lead Status`; `custom_current_program` links to `CRM Program`. The value must already exist in the target DocType, or the write is rejected.

**`Table` = a child table (rows inside a record).** A `Table` field holds a *list of sub-rows* that belong to the parent document. They have no independent life ‚Äî they live and die with the parent.

In this CRM, the lead carries identity + contact + routing on the parent, and **everything program-specific lives in child "profile" tables**:

```json
{
  "first_name": "Asha Verma",
  "mobile_no": "+919800000099",
  "custom_lab_profile": [
    { "hba1c": 7.2, "fbs": 138, "report_date": "2026-05-18" }
  ]
}
```

`custom_lab_profile` is a child table ‚Äî a list of rows. A lead can carry several profiles (Lab, Plan, Acquisition, Drug Program, Care & Providers); which fields YOUR key may send is whatever your key is granted ‚Äî call [`lead_schema`](/discover-schema) to see them.

## Writing child rows

When you send a child array, the API does **not** replace the existing rows. It **merges** by each row's **natural key**:

- **New key** ‚Üí a new row is appended.
- **Same key** ‚Üí that row is updated, and **only the fields you sent change** (omitted fields are left alone).
- **Rows you don't mention** ‚Üí untouched.
- **Delete one row** ‚Üí send `{ "<key_field>": <value>, "_delete": true }`.

This means a partial write ‚Äî say, just an updated `hba1c` for an existing report date ‚Äî never wipes the rest of that row, and never duplicates it.

### Row keys

A **multi-row** child must carry its **key field** on every row you send ‚Äî it's the row's address. The key is **required** even if your key wasn't granted it (the schema marks it `required`). Today's keys:

| Child table | Key field (required) | Rows |
|---|---|---|
| `custom_lab_profile` | `report_date` | Multi-row ‚Äî one per lab report date. |
| `custom_plan_profile` | `program_start_date` | Multi-row ‚Äî one per program start. |
| `custom_drug_program_profile` | `cycle_date` | Multi-row ‚Äî one per treatment cycle. |
| `custom_acquisition_profile` | ‚Äî | Single-row ‚Äî merges onto the one row. |
| `custom_care_providers_profile` | ‚Äî | Single-row ‚Äî merges onto the one row. |

Whether a child is multi-row (and which field is its key) is **discoverable** ‚Äî `lead_schema` returns `children.<table>.multi_row` and `children.<table>.key_field`. Read it at runtime rather than hardcoding.

```json
// Upsert: update the 2026-05-18 lab row's hba1c, ADD a new row for 2026-06-20,
// and DELETE the 2026-01-10 row ‚Äî all in one write. Other rows untouched.
{
  "custom_lab_profile": [
    { "report_date": "2026-05-18", "hba1c": 6.9 },
    { "report_date": "2026-06-20", "hba1c": 6.4, "fbs": 110 },
    { "report_date": "2026-01-10", "_delete": true }
  ]
}
```

When you read a lead back (`lead_get` / `lead_get_bulk`), each child row always carries its `name` (our internal row id) **and** its key field ‚Äî so you can address it on the next write.

## In practice

- You **cannot** create a child row on its own ‚Äî there's no `/api/resource/...` you POST a profile to (direct child-table access returns 403, by design). You send child rows *embedded in the parent* lead, exactly as above.
- A `Link` value must exist first. `custom_current_program: "Program B"` only works if `Program B` is a row in `CRM Program` ‚Äî check via [Schema discovery](/frappe-discovery).

‚Üí **Try it:** [Create a lead](/api/leads) ‚Äî its examples show profile child rows embedded in the body.
