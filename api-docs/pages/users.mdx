---
title: User onboarding
---

Every account in the CRM — a human agent or a machine integration — is a **`User`**, and its **roles** decide what it can do. This page covers the two admin jobs: onboarding an account, and minting an API key for an integration.

## Users are records

Read and manage users over the same REST API as everything else:

- **List** users to resolve a valid `lead_owner` before assigning a lead.
- **Read** a user to see its `roles` child table — the source of its capabilities.
- **Create** a user with roles inline (roles are a child table, so you set them in the same call).
- **Update** a user to disable it or change its roles.

The built-in `Administrator` account is specially protected and returns `403` on read; ordinary users read fine.

## Onboarding a person

One call creates the account and assigns its roles; `send_welcome_email: 1` triggers Frappe's set-your-password email.

- Create the `User` with `email`, `first_name`, the role(s) they need, and `send_welcome_email: 1`.
- To scope them to one group or program, add a [User Permission](/api/access-control#create-user-permission) afterwards.

## Onboarding an integration

This is exactly how the `api_key:api_secret` pair in [Authentication](/api/authentication) is produced:

1. **Create** a dedicated user (e.g. `you@yourcompany.com`) with the role its writes need — an admin role if it must set the locked routing fields.
2. **Generate keys** for that user — the response returns the `api_secret` **once**.
3. **Hand** the `api_key:api_secret` pair to the integration as a single secret.

> **⚠ `generate_keys` rotates.** Calling it invalidates the user's previous `api_secret` immediately, and the new secret is shown **only in that one response**. Capture it then — if you lose it you must regenerate, which breaks the old one.

## One key per integration

Give each integration its own user + key — never share. That way a key can be revoked, rotated, or audited per integration without touching the others.
